# 2026-07-12：WP-001 无引擎规则原型

## 这轮要证明什么

在创建 Godot 工程前，先把最容易产生歧义的规则写成可运行例子：道路接口怎样旋转、什么放置算非法、两种牌组怎样循环、同一随机局能否重放。

这轮没有画面、角色动画、触摸操作或正式关卡生成。

## 做成了什么

- 五种原型道路：直路、上坡、下坡、转角、桥。
- 左/上/右/下接口和顺时针旋转。
- 棋盘边界、占用、相邻接口、孤立放置和连通搜索。
- 四种明确失败原因：越界、重叠、接口不匹配、孤立。
- 固定顺序无限循环牌组。
- 弃牌堆洗回牌库，并用整数种子保证结果可复现。
- 一条平路、一条爬坡路线和一个错误接口失败样例。

## 怎样验证

运行 `npm test`：15 个测试全部通过。

运行 `npm run simulate:rules`：

```text
flat_success：WIN，第 3 回合
climb_success：WIN，第 3 回合
broken_failure：PLACEMENT_FAILED，第 1 回合，原因 PORT_MISMATCH
```

Node 测试运行器按文件自动发现时会创建子进程，第一次运行被受限环境拦截。这不是规则缺陷。随后增加单进程测试入口，日常 `npm test` 已可在普通权限下通过。

## 这轮得到的设计结论

- “一侧开口、另一侧封闭”应在放置时立即拒绝，不能等角色走到那里才告诉玩家。
- 允许接口暂时朝向空格，因为玩家需要从已铺道路继续向前施工。
- 胜负判断必须来自规则层的连通关系，不能靠画面看起来像接上了。
- 两种牌组循环都值得带进可玩原型：固定循环更适合计划，洗回更像传统卡牌 Roguelite，尚不能只凭代码决定哪种更有趣。
