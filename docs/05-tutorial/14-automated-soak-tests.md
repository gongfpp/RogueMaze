# 第 14 章：让游戏重复通关 250 次，有什么用

读完这一章，你能运行 RogueMaze 的自动耐久测试，知道失败输出该从哪里查，也不会把“程序跑了很多次”误写成“游戏已经好玩”。

## 先认识两个词

**耐久测试（soak test）**就是让同一套流程重复很多次，观察问题会不会累积。比如第一局有 8 张牌，领奖后有 9 张；重新开始时必须回到 8 张。如果重置漏了一步，短测试可能看不出来，第几十轮就会暴露。

**不变量（invariant）**是流程怎么变化都必须成立的事实。本项目当前检查的例子是：手牌始终为 4 张、每张卡的实例 ID 不重复、卡牌等级不超过上限。

## 一条命令运行

Windows：

```powershell
.\scripts\soak_test.ps1
```

Linux：

```sh
./scripts/soak_test.sh
```

默认是 250 次完整五节点远征，外加每轮一次失败流程。想做更重的本地检查，可以运行：

```powershell
.\scripts\soak_test.ps1 -Runs 1000
```

成功时会看到类似：

```text
Godot soak: 250 expedition(s), 36503 invariant check(s), all passed in 1.10s
```

## 脚本实际做了什么

1. 重置同一个 `GameSession`，确认节点、生命和牌组回到起点。
2. 铺一条平路跑完五个节点，在节点间轮换三种奖励。
3. 每次奖励后检查手牌、牌组、等级和实例 ID。
4. 再开一局故意不铺路，检查暂停、失败原因和失败事件。
5. 重复以上流程；任意检查失败就以非零退出码结束，让 CI 变红。

脚本暂时清空陷阱，因为这一轮想单独观察远征和重置。尖刺、蒸汽、落石、维修台各自已有更聚焦的测试。把所有系统都塞进一个大测试，失败时反而很难定位。

## 失败后怎么查

错误会包含轮次、节点和期望值，例如：

```text
run 73 node 2 keeps unique card instances | expected=9 actual=8
```

先看最早的一条错误，不要从最后一条开始猜。上例说明牌组有 9 张牌，但只有 8 个不同实例 ID，应优先检查创建卡牌和重置实例编号的代码。

修复后先用较小轮数快速复现，再跑全量：

```powershell
.\scripts\soak_test.ps1 -Runs 80
.\scripts\test_all.ps1
```

## 它不能证明什么

- 不能证明玩法有趣，也不能评价教学是否看得懂。
- 不能代替手机触摸误触、安全区和切后台测试。
- 快速规则循环没有持续渲染 20 分钟，所以不能单独证明帧率、显存和温度稳定。
- 250 次只是当前便宜且稳定的门禁，不是永远正确的神奇数字。

交付时应写“250 次状态机耐久通过”，不要写“完整质量验证通过”。准确描述证据，比堆漂亮术语更能帮助下一位成员。
