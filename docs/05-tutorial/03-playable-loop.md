# 第 03 章：把规则接成第一局能玩的游戏

这一章不要求你熟悉游戏架构。我们用当前灰盒解释五个基础概念：状态、每帧更新时间、输入映射、表现层，以及为什么“单个模块测试通过”还不等于游戏能玩。

## 1. 自动行进不是让坐标一直加一

角色不能无条件向右，因为道路可能上坡、下坡或断开。`RunnerState` 每到一格会做下面的事：

1. 读取当前道路有哪些接口。
2. 查找相邻格是否有反方向接口。
3. 排除刚刚走过的格子，避免立即掉头。
4. 找到目标后，在当前格与目标格之间逐渐移动。
5. 找不到就等待；宽限结束仍没路才失败。

角色位置、目标格、移动进度和是否失败合在一起叫“状态”。状态是程序此刻需要记住的事实。

## 2. `delta` 为什么重要

游戏每帧调用 `update(delta)`。`delta` 是上一帧到这一帧经过的秒数。

如果每帧固定移动 1 个像素，60 FPS 的电脑会比 30 FPS 的手机快一倍。现在使用：

```text
移动进度 += 每秒速度 × delta
```

这样设备帧率不同，角色用掉的现实时间仍大致相同。倒计时和断路宽限也使用 delta。

## 3. 一局游戏由谁管理

`GameSession` 像桌游主持人，持有：

- 棋盘和已经铺的道路。
- 自动行进角色。
- 牌库和四张手牌。
- 当前选中的牌与旋转角度。
- 倒计时、暂停、胜利或失败。

表现层不直接修改这些变量。它说“选择第 2 张牌”或“尝试放到 `(3, 2)`”，GameSession 返回成功或具体失败原因。

这让触屏、鼠标和键盘可以共用规则。

## 4. 点击和拖拽为什么不能写两套规则

手机玩家习惯拖卡，桌面玩家可能更喜欢先点击牌再点击格子。两种操作最后都调用 `place_selected`：

```text
触摸拖放 ─┐
鼠标点击 ─┼→ place_selected → 棋盘校验 → 补牌
键盘选择 ─┘
```

如果拖拽直接创建道路、点击却经过棋盘校验，就会出现“同一格拖得进去但点不进去”的错误。

## 5. 灰盒为什么只画几何图形

灰盒用方格、线、圆和颜色表达功能：

- 黄色线是普通道路。
- 青色线是桥。
- 白色圆是角色。
- 绿色格是当前牌的合法位置。
- 拖到非法格会显示红色。

现在验证的是信息是否清楚，不是美术是否精致。如果绿色格仍看不懂，换成漂亮贴图不会自动解决问题。

## 6. 从单元测试到完整单局

我们已经分别测试道路、牌组和角色，但还需要集成测试：自动从第 1 格铺到第 7 格，然后让角色跑到终点。

这能发现模块连接问题，例如：

- 放牌成功却没有补牌。
- 道路看似连通，角色却读不到。
- 角色到终点但 GameSession 没变成胜利。

当前完整单局和其他规则合计 58 项 Godot 断言。

## 7. 程序截图能检查什么

`tests/godot/capture_ui.gd` 会用 405×720 窗口加载真实主场景并保存截图。它帮助检查：

- 控件是否跑出屏幕。
- 棋盘和卡牌是否互相遮挡。
- 手牌是否处于底部拇指区。
- 颜色和文字是否有基本层级。

截图不能替代真人试玩，因为它不知道按钮是否好点、节奏是否紧张。

## 8. 新成员怎样交付一个交互模块

以“增加双击旋转”为例：

1. 说明它要解决的真实操作问题。
2. 只在输入层识别双击，仍调用现有 `rotate_selected`。
3. 补充输入测试或手工复现步骤。
4. 分别检查触屏和鼠标没有被破坏。
5. 更新教程、日志和 commit。

不要在输入脚本里顺便修改道路接口。那会让模块边界失去意义，也让回归范围变大。
