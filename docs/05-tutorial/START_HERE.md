# 零基础成员从这里开始

这篇不是考试大纲。读完后，你应该能回答三件事：游戏程序大致怎样运转、Git 为什么能保护我们的工作、接到一个小模块后怎样把它做完并交出去。

## 1. 游戏程序其实在反复做三件事

游戏运行时，会非常快地重复下面三步：

1. **接收输入**：玩家点了一张牌、拖动鼠标，或者按下旋转键。
2. **更新状态**：程序判断这张路能不能放，角色走到了哪里，是否受伤。
3. **画出结果**：把新道路、角色位置、动画和提示显示出来。

一次重复常被叫作“一帧”。60 FPS 的意思是理想情况下每秒完成约 60 帧，不是说所有逻辑都必须塞进同一段代码。

在 RogueMaze 里，玩家把上坡牌拖到格子上属于输入；判断左右接口能否接上属于状态更新；绿色落点和拼接动画属于画面反馈。

## 2. 常见名词，先用人话理解

### 场景（Scene）

场景是一组能一起出现、一起复用的游戏对象。主菜单可以是一个场景，一张道路也可以是一个小场景。它像舞台布景，不一定等于完整关卡。

### 节点（Node）

节点是 Godot 场景里的一个组成部分。图片、按钮、声音播放器都可以是节点。多个节点像积木一样组成场景。

### 脚本（Script）

脚本是写给对象的行为说明。例如“被点击时旋转 90 度”“生命变成 0 时发出失败消息”。本项目暂定使用 GDScript，它的写法接近简化的 Python，但两者不是同一种语言。

### 数据和状态

“直路牌的左右有接口”是比较固定的数据；“这张牌现在位于手牌第 2 格”是运行中的状态。把两者分清，能避免改一张牌时意外影响所有牌。

### 事件（Event）

事件就是“某件事已经发生”的通知，例如 `road_placed`（道路已放置）。音效、动画和任务统计都可以听这个通知，不需要把代码互相缠死。

### 随机种子（Seed）

电脑随机通常是按公式生成的。给它同一个起始数字，也就是种子，就能重现相同牌序。这样玩家报告“第 5 张牌必死”时，开发者能真的复现。

### 测试

测试不是最后点几下看看。它是一个可重复的问题，例如：“直路旋转 90 度后，接口是否从左右变成上下？”自动测试会运行这个问题并比较实际答案。

### 构建或导出（Build/Export）

编辑器里的项目不能直接交给玩家。导出会把代码和资源整理成 Windows 程序、Linux 程序、Android 安装包或 iOS 工程。不同平台共用玩法，但签名、文件路径和输入方式可能不同。

## 3. Git 到底保护什么

Git 像一套带说明的项目存档，但比“最终版2_真的最终版.zip”可靠。

- **仓库（repository）**：被 Git 管理的整个项目目录。
- **工作区（working tree）**：你现在正在修改、还没存进版本历史的文件。
- **commit**：一次有说明、可比较、可回到的项目快照。
- **分支（branch）**：一条独立修改路线，让功能开发不立刻影响稳定版本。
- **远端（remote）**：网络上的仓库。本项目的 `origin` 是 GitHub 上的 `gongfpp/RogueMaze`。
- **push**：把本地已有的 commit 上传到远端。它不会自动把没 commit 的文件传上去。

本项目每轮都 commit，是为了让错误容易定位和回退。好的 commit 不追求文件越多越好，而是让别人能看懂“这次完成了什么，并且它已经怎样验证”。

## 4. 一个模块怎样从领取到交付

假设你领取“直路旋转”这个任务：

1. 先读工作包，确认输入、输出和明确不做的内容。
2. 找到相关数据和测试，先运行现有测试，确认起点是正常的。
3. 做最小修改，让直路的左右接口旋转后变成上下。
4. 增加或更新自动测试，再运行一次。
5. 实际打开调试场景，看画面是否和规则一致。
6. 更新开发日志和教程中受影响的部分。
7. 查看 Git 变化，只提交本任务文件，写清 commit 说明并推送。

如果做着做着发现还需要重写牌库，不要偷偷扩大任务。记录依赖，把它拆成另一个工作包。可交付的小模块比“大概完成了八成”的大模块更容易协作。

## 5. 不会写代码也能参与什么

- 按试玩模板记录第一次困惑的位置，不替测试者解释。
- 制作道路图标草稿，并按照资产命名和尺寸要求交付源文件。
- 根据验收步骤在不同电脑或手机上测试构建。
- 校对教程：照着步骤做，标出跳步、术语和无法复现的结果。
- 整理卡牌表格、陷阱规则或关卡样例，但不要私自改变已经锁定的规则。

这些都是真实的项目工作。关键不是会多少术语，而是输入清楚、结果可检查、文件能交给下一位成员。

## 6. 提交模块前的五个问题

- 我完成的是哪个工作包？
- 别人怎样运行或看到结果？
- 我用什么证明它做对了？
- 是否混入了与任务无关的修改？
- 日志、教程和 commit 说明能让下一位成员接手吗？

五个问题都能回答，再交付。
