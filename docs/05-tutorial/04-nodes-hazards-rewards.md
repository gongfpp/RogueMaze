# 第 04 章：怎样把一局扩成三个有变化的节点

这一章讲三个常见游戏开发概念：流程状态、跨关保留的数据，以及怎样让陷阱“给玩家机会”，而不是突然判死刑。

## 1. 节点不是复制三份场景

最直接的做法是做 `Level1.tscn`、`Level2.tscn`、`Level3.tscn`，每份都复制棋盘和手牌。短期看很快，修改一次按钮却要改三份。

当前实现只保留一个 GameSession，通过 `node_index` 决定本节点有哪些危险：

```text
节点 1：基础道路
节点 2：基础道路 + 尖刺
节点 3：基础道路 + 尖刺 + 落石
```

棋盘和角色每节点重新创建，生命与本次远征牌组继续保留。这就是“节点内状态”和“跨节点状态”的区别。

## 2. 游戏流程也是状态

除了角色会 RUNNING/WAITING，整局也有：

- `PLAYING`：能放牌，角色会移动。
- `REWARD`：角色停下，玩家选择奖励。
- `WON`：三个节点全部通过。
- `LOST`：断路、落石或生命耗尽。

输入层先看当前状态。例如 REWARD 时点击卡片应该选择奖励，不能同时把道路放进棋盘。

## 3. 尖刺为什么只伤一次

尖刺保存一个 `spent` 标记。角色第一次进入扣血，之后不重复扣。这里的目的不是模拟真实尖刺，而是让同一格的结果可预测。

如果未来设计需要持续伤害，要明确写成新规则，并在视觉上显示“站在这里会继续受伤”，不能只改数字。

## 4. 落石为什么有倒计时

落石会改变已经铺好的道路，风险比尖刺更大。为了公平，它在格子上方显示岩石和剩余秒数。计时结束时：

1. 标记落石已触发。
2. 移除对应道路。
3. 如果角色正在该格或正向该格移动，立即失败。
4. 否则角色以后走到断口时按普通断路处理。

“预警 + 两种应对”是陷阱设计的实用检查：玩家可以绕路，也可以等落石后再铺。

## 5. 卡牌奖励怎样跨节点

`run_deck` 是本次远征的牌表。进入节点时，DeckState 用它建立循环牌库；选择奖励会把一张道路 ID 追加进去。

我们没有直接把奖励塞进当前手牌，因为节点已经结束。下一节点重新发牌，流程更清楚。后续若希望奖励立即可用，可以明确设计“加入手牌”奖励，不能让所有奖励隐式改变规则。

## 6. 为什么失败原因属于应用层

角色只知道自己因没有下一块道路而 FAILED。GameSession 还知道落石是否刚好砸中、生命是否归零，因此由它记录：

- `ROAD_MISSING`
- `ROCK_HIT`
- `NO_HEALTH`

界面再把代码原因翻译成玩家能理解的句子。以后做多语言时，只换文字，不换规则。

## 7. 怎样测试三节点而不等几十秒

测试可以直接把倒计时设为 0，并用较大的 delta 推进角色。它仍调用正式更新逻辑，只是不按现实时间等待。

完整测试依次检查奖励、牌组、节点、扣血和总胜利；落石测试把计时设为 0.01 秒，快速确认道路被删除。

测试加速不是作弊。只要输入和正式逻辑相同，它就是让回归更快。

## 8. 模块交付检查

如果你负责新增“腐蚀雾”陷阱，交付至少包含：

- 什么时候预警、什么时候生效。
- 玩家至少两种应对办法。
- 陷阱数据放在哪个层级。
- 角色/道路受到影响的自动测试。
- 一张窄屏截图和一次失败原因检查。

只有画了绿色烟雾但没有规则、反馈和测试，不算完成一个陷阱模块。
