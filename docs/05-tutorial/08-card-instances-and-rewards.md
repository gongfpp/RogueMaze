# 第 08 章：同名的两张牌，也可能不是同一张牌

这一章解释游戏开发里很常见的“定义”和“实例”。不需要先懂面向对象；先从一副真实纸牌想起。

## 1. 牌名不等于那张牌

“直路”是牌的种类，类似扑克牌里的“红桃”。玩家牌组里可以有四张直路，其中一张强化过，另外三张没有。只保存 `straight` 这个名字，就无法回答“强化的是哪一张”。

因此项目新增 `CardState`：

```text
实例 17：straight，等级 2
实例 18：straight，等级 1
```

道路定义仍回答“直路有哪些接口”；卡牌实例回答“玩家拥有的这一张直路现在是什么状态”。

## 2. 为什么需要实例 ID

奖励界面打开时，玩家看到“升级一张直路”。从点击到应用奖励之间，程序必须始终指向同一张牌。数组下标不可靠，因为删牌、排序或洗牌都会改变位置。实例 ID 像一张不会重复的工牌，用来找到目标。

实例 ID 不一定要显示给玩家。它首先是程序和测试保持指向稳定的工具。

## 3. 加牌、升级、移除改变什么

- 加牌：选择面更多，但想再次抽到关键牌要等更久。
- 升级：牌组长度不变，强化一个明确工具。
- 移除：选择变少，却能让精简组合更快循环。

这三种动作才形成取舍。如果升级只是把数字从 1 改成 2，却不影响任何危险，它仍然是假选择。本项目让 2 级道路吸收一次落石，并在画面上用 `+`、`ARMOR` 和道路外圈表达。

## 4. 状态应该由谁决定

落石被护甲挡住后，需要同时发生：道路保留并降级、落石不再伤人、界面显示已解除、播放反馈。

正确的数据流是：

```text
GameSession 判定道路挡住落石
    → 写入 neutralized 状态并发出 ROAD_SAVED 事件
    → UI 读取状态画图，读取事件播放反馈
```

不要让 UI 自己猜“路还在，所以落石大概被挡住了”。规则结论放在玩法层，画面只是翻译结论。

## 5. 最小测试怎么写

先测小对象：新牌是 1 级、只能升到 2 级、升级后后缀是 `+`。再测奖励：加牌后数量 +1，升级后数量不变且目标实例变 2 级，移除后数量 -1 且找不到目标 ID。

最后做集成测试：在落石坐标放一张 2 级道路，让倒计时结束，确认道路仍在、等级回到 1、落石状态为已解除。

小测试告诉你“零件正确”，集成测试告诉你“零件接起来仍然正确”。两种都需要。

## 6. 给新成员的可交付任务

可以领取“为强化牌增加另一种清晰反馈”，但要把范围写清：不改变强化规则，只调整图形、颜色或短音；保留减少动态效果选项；补一张 405×720 截图；运行全量测试。

交付说明要回答：普通牌和强化牌能否一眼区分？色弱或静音玩家是否仍能看懂？缩到手机尺寸后文字是否还在牌内？
