# 第 06 章：存档不是把变量随便写进文件

这章给第一次做本地存档的成员。你会理解 `user://`、数据版本、校验、迁移和原子替换分别解决什么问题。

## 1. 什么应该保存

当前保存两类数据：

- 设置：SFX 是否开启、是否减少动态效果。
- 进度：远征胜场、最佳节点、解锁的道路。

角色正走到半格、当前动画时间等临时状态不保存。保存越多不是越完善；只有玩家真正期望恢复的状态才进入格式。

## 2. 为什么使用 `user://`

导出后的游戏资源通常只读。Godot 的 `user://` 会指向每个平台允许应用写入的位置：Windows、Linux、Android、iOS 路径不同，但代码不用分别硬编码。

开发脚本把 APPDATA/XDG 临时指向 `.tools`，只是为了测试隔离。正式包仍使用系统标准位置。

## 3. 数据版本有什么用

设置 v1 长这样：

```json
{
  "version": 1,
  "sfx_enabled": true,
  "reduced_motion": false
}
```

以后 v2 增加音乐音量，加载器能知道旧文件没有该字段，并补默认值。没有版本仍按 v0 旧数据迁移，不能直接判坏档。

## 4. 永远不要相信文件里的值

文件可能来自旧版本、手工编辑或半次写入。加载后逐项检查：

- 布尔值必须真的是 bool。
- 胜场不能小于 0。
- 最佳节点不能超过 3。
- 道路 ID 必须存在于 RoadCatalog。
- 解锁列表要去重。

解析失败时回到默认值，游戏仍能启动。

## 5. 什么是原子替换

直接打开正式文件覆盖时，如果进程在写到一半崩溃，旧数据已经没了，新数据又不完整。

当前步骤：

```text
写 save.json.tmp 并 flush
旧 save.json → save.json.bak
save.json.tmp → save.json
成功后删除 .bak
```

如果第二次改名失败，尝试把备份恢复。这里的“原子”指正式文件在关键替换点前后都是完整版本，不是说任何硬件故障都不会丢数据。

## 6. 怎样测试存档

测试不能只调用 `sanitize`。还要真的：

1. 在 `user://tests` 写设置和进度。
2. 创建新的 PersistenceService，模拟重启。
3. 读回并比较字段。
4. 更新一次胜场再重读。
5. 写入损坏 JSON，确认安全回退。
6. 清理测试的正式、临时和备份文件。

测试路径和玩家路径分开，失败的测试不会毁掉开发者存档。

## 7. 为什么退出码 0 仍可能失败

一次调试中，测试脚本发生运行时错误，但 Godot 进程最后仍返回 0。只检查 `$LASTEXITCODE` 会误报通过。

现在测试包装脚本同时要求：

- 没有 `SCRIPT ERROR`。
- 没有脚本加载失败。
- 输出明确包含断言全部通过标记。

自动化的职责不是“尽量变绿”，而是让绿色可信。

## 8. 模块交付

新增存档字段时必须一起交付：默认值、类型/范围校验、旧版本迁移、往返测试、损坏文件行为和面向玩家的可见入口。只在 JSON 里多写一个字段不算完成。
