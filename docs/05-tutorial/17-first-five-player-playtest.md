# 第 17 章：第一次把游戏交给 5 个陌生玩家

读完这一章，你能生成一个不缺文件的试玩包，主持一次不诱导答案的 20 分钟测试，并把观察交给策划、程序或 UI 同事继续处理。

## “我自己玩通了”为什么不够

开发者已经知道绿色格、道路端口和 ARMOR 的含义，大脑会自动补齐游戏没讲清楚的部分。陌生玩家没有这些背景。5 人试玩不是用很小样本证明游戏一定好玩，而是尽早发现重复出现的理解障碍。

5 不是神奇统计数字。它只是当前小团队负担得起、又能看到一些重复模式的起点。

## 先生成可交付的包

Windows：

```powershell
.\scripts\package_windows_playtest.ps1
```

脚本会先测试和构建，再在 `builds/playtest` 生成 ZIP。不要从 `builds/windows` 随手拖一个 EXE 发出去，因为那样会漏掉说明、Build 身份和校验值。

ZIP 必须包含：

- `RogueMaze.exe`
- `PLAYER_README.txt`
- `BUILD.txt`
- `SHA256SUMS.txt`

`SHA256` 可以理解为文件指纹。玩家算出的指纹与团队提供的一致，说明传输后文件没有变化；它不能替代代码签名，也不能保证程序天然安全。

## 为什么 dirty 包不能当正式批次

Build 后面出现 `*`，表示导出时有未提交修改。这样的包适合验证打包脚本，但团队之后很难准确还原源码。正式 5 人批次应该：

1. 工作树干净。
2. 全量测试通过。
3. commit 已推送，Actions 全绿。
4. 再运行打包脚本。
5. 在观察表中抄同一个 Build。

## 主持人最难的工作是忍住提示

玩家把上坡牌放错时，不要立刻说“先旋转”。先记录他看到了什么、期待什么。如果每次卡住都由主持人解答，最终只能证明主持人会玩。

可以问：“你刚才期待发生什么？”不要问：“是不是旋转按钮太小？”后者已经把答案塞给玩家。

## 一场 20 分钟测试

- 2 分钟：核对包、Build、SHA-256和匿名记录同意。
- 8 分钟：首局，不讲核心解法。
- 3 分钟：三个固定问题。
- 5 分钟：第二次尝试，只回答明确的操作问题。
- 2 分钟：抄结算数据，确认是否愿意再玩。

完整话术见 `docs/06-playtest/PLAYTEST_GUIDE.md`，记录用 `OBSERVATION_FORM.md`。

## 怎样把观察变成工作

不要把每句话都变成需求。先按根因合并：3 人都没发现旋转，可能是一项“旋转入口不可见”，不是 3 个缺陷。

- P0：无法开始、卡死、存档破坏、主流程必现崩溃。
- P1：能继续，但多数目标玩家明显困惑或频繁误操作。
- P2：偏好、润色或低频建议。

报告要同时保留行为与原话。`MISPLAYS 9` 是行为，“我以为红色也能放，只是会扣血”解释了原因。只看其中一半都容易误判。

最后，准备好 ZIP 和表格仍不叫“完成 5 人试玩”。只有真实执行、问题归类、P0 回归以及 GDD/Backlog 更新后，门禁才通过。
