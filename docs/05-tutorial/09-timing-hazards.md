# 第 09 章：做一个会开关的陷阱

尖刺只问“走不走这格”，落石主要问“能不能赶在倒计时前”。蒸汽喷口再加入一个问题：“现在是不是安全窗口？”这一章用它解释周期、相位和状态反馈。

## 1. 周期是什么

蒸汽每 4 秒重复一次，其中前 1.4 秒开启，后 2.6 秒关闭。4 秒就是周期，1.4 秒是开启时长。

把已经过去的时间除以周期，只关心余数：

```text
周期内时间 = (经过时间 + 初始相位) % 周期
周期内时间 < 开启时长  →  喷口开启
```

`%` 是取余数。经过 5 秒，在 4 秒周期里相当于新一轮的第 1 秒。

## 2. 相位为什么有用

如果一排三个喷口都从第 0 秒一起开启，它们看起来像同一个大陷阱。给不同喷口设置相位，可以让它们错峰开关，形成观察和选路。

相位不是另一个计时器，只是计算时给时间加一个偏移。这样同一套规则可以组合出同步、交替或追逐波。

## 3. 玩家需要看见状态

程序知道开启还不够。项目同时使用：

- 红色与蒸汽线表示危险；蓝色圆环表示关闭。
- 数字显示距离下一次开关还有多久。
- 受伤时显示文字并播放已有伤害提示音。

颜色只是其中一层，不能让色弱玩家只能猜。数字和形状也必须变化。

## 4. 数据和规则怎样分工

内容文件决定位置、周期、开启时长、相位和伤害。`GameSession` 决定何时开启、进入时是否扣血、生命归零是否失败。UI 调用同一个开启判定绘图，不复制一套“看起来差不多”的计时公式。

一旦 UI 和规则各写一套公式，就可能出现画面已变蓝、碰上去仍扣血的严重问题。

## 5. 自动测试不能只测公式

最小测试要覆盖：第 0 秒开启、进入扣 1 点生命并发事件；第 2 秒关闭、进入不扣血。完整流程还要经过五节点，确认新增节点不会让奖励或胜利状态卡住。

最后看手机尺寸截图：计时数字是否压住道路？开启与关闭形状是否可区分？这属于视觉验收，不是数学断言能回答的。

## 6. 关卡设计实习生的练习

只用现有三种陷阱设计一个新节点草案，不写新程序。写出：每个陷阱位置和参数、至少两条应对路线、最坏手牌是否仍可继续、玩家第一次失败后能学到什么。

先在纸上或表格验证，再改 `gameplay.json`。提交时附一次完整试玩记录，不只说“感觉还行”。
