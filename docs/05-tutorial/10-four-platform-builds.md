# 第 10 章：写了“四平台支持”，不等于四个平台都测试过

跨平台游戏的大部分规则可以共用，但“能导出”“能安装”“能启动”“能发布”是四个不同阶段。文档必须说清证据到哪一步，不能用一个 Windows 截图代表所有平台。

## 1. 导出预设是什么

导出预设是一组打包说明：目标平台、CPU 架构、包名、版本号、图标、权限和输出路径。它不包含游戏规则，也不应该包含密码。

本项目有 Windows、Linux、Android、iOS 四个预设。Android 只打 arm64 调试 APK；iOS 先导出 Xcode 工程，再由 Xcode 签名和安装。

## 2. 包名为什么不能随便改

`com.gongfpp.roguemaze` 是应用身份。商店和手机用它判断“这是原应用的更新，还是一款新应用”。一旦正式发布，再改包名就会变成新的安装和新的数据目录。

因此开发期可以暂定，首次商店建档前必须由发行负责人确认域名/组织命名和全球唯一性。

## 3. 为什么不把签名密码写进 Git

签名证明“这个包确实由发布者制作”。泄露签名材料相当于把盖章交出去。仓库只保存空字段和构建方式；CI 或发布电脑通过环境变量/密钥库提供 Team ID、证书、描述文件和 keystore 密码。

教程里的占位符也不能放真实密码。`.gitignore` 不是保险箱：文件一旦提交过，删除当前版本并不会自动删除历史。

## 4. Android 工具为什么没有自动静默安装

Android SDK 下载前要求接受许可协议。程序可以自动下载文件，但不能替团队成员声称“我有权代表公司接受协议”。本项目的安装脚本要求显式参数：

```powershell
.\scripts\setup_android_sdk.ps1 -AcceptAndroidSdkLicense
```

运行它的人应先阅读官方协议，并确认自己有权限接受。脚本随后校验下载摘要、安装锁定版本并更新本地 Godot 设置；`.tools` 不进入 Git。

## 5. iOS 为什么必须用 Mac

Godot 会生成 Xcode 工程，Xcode 再调用 Apple SDK 编译、签名、安装或上传。Xcode 只运行在 macOS。Windows 可以维护共用代码、预设和测试，却不能提供“iPhone 已通过”的证据。

macOS 发布机设置 `GODOT_IOS_TEAM_ID` 后运行 `scripts/build_ios.sh`。脚本临时写入 Team ID，结束时恢复空占位，避免把团队身份混进普通开发提交。

## 6. 新成员怎样报告平台状态

使用下面这些精确说法：

- “Windows release 导出成功并启动 20 分钟。”
- “Linux x86_64 从 Windows 模板导出成功，尚未在 Linux 启动。”
- “Android 预设可解析，SDK 未安装，未生成 APK。”
- “iOS Xcode 工程预设已提交，等待 macOS、Team ID 和真机。”

不要写“基本支持”“应该能跑”。状态越具体，下一位成员越知道从哪里接手。

还有一个常见坑：外部工具返回 0 不一定代表导出成功。本项目除了检查退出码，还检查日志中的 `Cannot export project`、`Project export failed` 和脚本编译错误。自动化的价值不是“命令跑过了”，而是能证明产物真的生成。
