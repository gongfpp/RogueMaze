# 第 01 章：先不做画面，把拼路规则跑通

这一章适合第一次接触游戏逻辑的实习生。你会运行一个没有画面的小游戏规则，理解方格坐标、道路接口、旋转、失败原因、随机种子和自动测试。最后你会知道这份原型怎样交给 Godot 模块的开发者。

## 1. 为什么“没有画面”也算原型

原型不是低清晰度的正式游戏，而是用最便宜的方法回答一个问题。

这轮的问题是：“两块道路到底什么时候算接上？”如果先做角色动画和漂亮的桥，规则改一次就要跟着改很多画面。现在只用文字和测试，改规则的成本很低。

可以把它想成桌游设计师先用纸片试规则。Node.js 在这里相当于一个不会算错、还能反复重放的桌游主持人。

## 2. 先运行，再读代码

在项目根目录打开终端：

```powershell
npm test
npm run simulate:rules
```

第一条命令应该显示 15 个测试通过。第二条会显示两个胜利样例和一个失败样例。如果命令不存在，先检查 `node --version`，本原型要求 Node.js 20 或更高版本。

不要急着逐行读代码。先看到可运行结果，才能知道每个文件在帮助什么。

## 3. 方格坐标是什么

棋盘左上角是 `(0, 0)`：

```text
       x →
    0   1   2
y  ┌───┬───┬───┐
↓ 0│0,0│1,0│2,0│
  1│0,1│1,1│2,1│
   └───┴───┴───┘
```

`x` 增加代表向右，`y` 增加代表向下。手机、Windows 和 Linux 都使用同一套逻辑坐标；屏幕像素多大不影响道路是否连接。

这就是“规则与画面分开”的第一个例子。

## 4. 用接口描述道路

直路不是一张图片，而是“左边和右边有出口”这条数据：

```text
直路：LEFT + RIGHT       ───
上坡：LEFT + UP          ──┘
```

两个相邻格子要连接，双方必须同时朝向对方开口。左边道路有 `RIGHT`，右边道路就必须有 `LEFT`。只有一边开口会返回 `PORT_MISMATCH`，也就是“接口不匹配”。

这种写法的好处是，美术以后把路画成纸桥、木桥或机械管道，连接规则都不用重写。

## 5. 旋转不是换一张图片

代码把顺时针 90 度看作一次旋转：

```text
LEFT → UP → RIGHT → DOWN → LEFT
```

一张左右直路旋转一次，就变成上下直路。旋转四次回到原状。测试会检查这个规律，所以未来即使换了实现方式，也不能悄悄把方向转反。

## 6. 为什么失败原因要分开

当前原型区分四种失败：

- `OUT_OF_BOUNDS`：放到棋盘外。
- `OCCUPIED`：格子已经有路。
- `PORT_MISMATCH`：相邻道路只有一侧开口。
- `ISOLATED`：新路没有接上任何已有道路。

如果程序只回答“失败”，界面就只能弹一个模糊红叉。分开原因后，界面可以显示“这里已有道路”或突出那个没接上的接口，玩家才知道怎样改。

## 7. 两种牌组循环

固定循环会一直保持作者写下的顺序：

```text
A, B, C, A, B, C ...
```

它适合谜题和计划型玩法。玩家记住牌序后，可以提前安排路线。

弃牌洗回则在牌库用完后，把已用过的牌重新打乱。它更接近常见卡牌 Roguelite。这里的“随机”由一个整数种子开始；同一种子会得到同一序列，方便复现玩家遇到的牌局。

代码能证明两种方案都稳定，却不能证明哪一种更好玩。这个问题要留给后面的可玩测试。

## 8. 自动测试怎样读

测试通常有三段：准备条件、执行动作、检查结果。例如：

1. 准备一块已有直路。
2. 在它右边放一块旋转成竖直的直路。
3. 检查结果是不是 `PORT_MISMATCH`。

测试不是为了证明开发者永远不会犯错。它是把已经同意的规则变成警报器：以后有人改旋转代码，旧规则被破坏时会立刻看到失败。

## 9. 文件怎样分工

- `road.js`：方向、道路类型和旋转。适合规则/数据成员维护。
- `board.js`：放置和连通。适合核心玩法成员维护。
- `deck.js`：牌序和随机。适合卡牌系统成员维护。
- `simulator.js`：把以上部分串成一局。适合集成开发者维护。
- `test/`：任何成员修改对应模块时一起维护。

边界不是说“别人不能改”，而是说明修改时首先找谁确认、交付时通知谁复查。

## 10. 交给 Godot 开发者什么

Node.js 代码不会放进正式游戏包。真正要交接的是：

- 四方向接口的含义。
- 旋转的顺序。
- 四种失败原因。
- 三个胜负样例。
- 15 个测试所表达的规则。

Godot 版本可以使用 GDScript 重写，但同样的输入必须得到同样的结果。这种“可运行的规则说明”比一句“照着做就行”更容易验收。

## 11. 你可以做的小练习

给 `board.test.js` 增加一个测试：一块道路放在已经占用的格子时，应返回 `OCCUPIED`。项目里已经有答案，可以先自己写，再对照现有测试。

交付时说明：改了哪个测试、怎样运行、看到什么结果。不要顺手增加新道路或改牌组规则，那属于新的工作包。
