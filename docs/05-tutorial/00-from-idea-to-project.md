# 第 00 章：把游戏点子变成可执行项目

## 1. 为什么先不打开引擎

“小人自动走，玩家拼路”是一个很好的体验钩子，但它还不是可测试的规则。直接做画面会立刻遇到一连串互相影响的问题：道路按格还是自由摆？卡牌是否旋转？抽完如何循环？没有路时立刻死还是等待？如果这些没有边界，代码很快就会把临时猜测固化。

预制作的任务不是把所有问题想完，而是把最危险的未知问题排到最前面。

## 2. 从一句话提炼设计支柱

本项目的一句话是：玩家在自动前进的时间压力下，用循环道路卡牌创造生路。

从中可提炼四个不可随意丢失的部分：

- 自动前进带来时间压力。
- 道路是玩家的主要输入，不直接控制角色。
- 手牌有限并循环，产生预判和构筑。
- 高低差、陷阱和奖励让“能通”之外还有路线取舍。

这就是设计支柱的用处：以后出现“要不要加攻击按钮”时，可以判断它是否强化核心体验，而不是只判断它听起来是否有趣。

## 3. MVP 是一个问题，不是一张功能清单

RogueMaze 的 MVP 要回答：这套核心循环能否让玩家连续玩几局并做出不同决策？因此，MVP 需要胜负、一次构筑和少量陷阱，却不需要排行榜、公会或五套角色。

一个实用做法是把范围分成：

- Must：不做就无法回答核心问题。
- Should：明显改善验证质量，但必要时可推迟。
- Could：有价值，但不能阻塞里程碑。
- Won't：这次明确不做，防止反复讨论。

“Won't”不是永远不做，而是保护当前实验。

## 4. 把路线图拆成可验收工作包

“完成卡牌系统”仍然太大。一个好的工作包应能独立回答一个问题，例如：

- 给定固定随机种子，弃牌洗回是否可复现？
- 相邻道路的两个端口怎样才算连接？
- 角色走到断口时，规则层返回什么失败原因？

每个工作包都写范围外事项，可以有效阻止开发中顺手扩张。

## 5. 计划、日志和决策记录的区别

- 计划说明准备做什么，允许随新证据调整。
- 日志说明实际做了什么，是事实记录。
- ADR 说明为什么选择某个有长期影响的方案，以及何时重新评估。

三者分开后，项目不会因为修改计划而丢失历史，也不会把过时日志误当成当前设计。

## 6. 本项目的第一次风险排序

最危险的不是“有没有足够多的卡牌”，而是：实时触屏放路是否舒服、随机循环是否公平、规则是否能稳定复现。因此下一工作包是纯规则原型，而不是制作主菜单或角色动画。

这是一条很通用的开发原则：**先验证会让整个项目失去成立理由的假设。**
