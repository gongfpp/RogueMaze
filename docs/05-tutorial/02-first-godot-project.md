# 第 02 章：第一次把规则放进 Godot

这章写给第一次打开游戏引擎的人。你会知道 Godot 在项目里负责什么、为什么规则脚本不直接控制按钮，以及怎样运行测试和导出桌面程序。

## 1. 引擎不是“帮我们自动做完游戏”

Godot 提供窗口、画面、声音、输入、场景编辑器和平台导出。它不会替我们决定道路怎样连接，也不知道哪张卡牌应该进入弃牌堆。

所以项目分成两类代码：

- `game/domain` 只回答规则问题，例如“能不能放”和“是否连通”。
- `game/presentation` 把规则画出来，并接收触摸、鼠标和键盘。

这样更换按钮样式不会破坏牌组，更换道路图片也不会改变胜负。

## 2. 项目文件和主场景

`project.godot` 是项目的总设置，里面记录游戏名、默认窗口、渲染方式、输入动作和第一个场景。

当前主场景是 `game/presentation/main.tscn`。`.tscn` 是文本格式的场景文件，描述标题、面板、按钮等节点。`main.gd` 是它的行为脚本，点击按钮时运行三条规则样例。

## 3. 为什么选 Compatibility 渲染

渲染器负责把场景变成屏幕像素。RogueMaze 是轻量 2D 游戏，比最新光照效果更需要覆盖普通手机和旧一些的桌面显卡。Compatibility 使用 OpenGL 路线，作为当前最稳妥的覆盖基线。

这不是永远不能改。若未来某种视觉效果只有 Mobile/Vulkan 能实现，需要先在低端设备测帧率，再写新的技术决策。

## 4. 运行全部测试

Windows：

```powershell
.\scripts\test_all.ps1
```

Linux：

```sh
./scripts/test_all.sh
```

测试分两组：Node 版本验证最初的规则规格；Godot 版本验证正式 GDScript 实现。两组都通过，说明移植没有改变已经同意的规则。

## 5. 无窗口模式是什么

自动测试不需要真的弹出游戏窗口。`--headless` 让 Godot 不创建图形窗口，只加载项目并运行脚本。这对服务器和 CI 很重要，也能减少测试受显卡驱动影响。

“无窗口”不等于“不测试画面”。布局、拖拽和动画仍需场景测试、截图和真机试玩；只是纯规则不必为等窗口浪费时间。

## 6. 自包含工具为什么不提交

Godot 编辑器和导出模板超过 1 GB，而且不同操作系统需要不同程序。它们放在 `.tools` 并被 Git 忽略：

- 仓库只保存版本号、下载说明和脚本。
- 每位开发者在本机准备对应工具。
- 游戏代码的 commit 不会被巨大二进制淹没。

`_sc_` 文件让 Godot 把编辑器设置放在自身旁边。开发脚本还把临时用户目录放进 `.tools`，公共电脑和受限环境就不会污染个人配置。

## 7. 导出和运行项目有什么区别

在编辑器运行时，Godot直接读取项目脚本和资源。导出时，它把这些内容打包，并和目标平台的运行模板组合：

- Windows 得到 `.exe`。
- Linux 得到 `.x86_64` ELF 程序。
- Android 通常得到 `.apk` 或商店使用的 `.aab`。
- iOS 先得到 Xcode 工程，再由 macOS/Xcode 签名构建。

当前 Windows 和 Linux 已实际生成；Android/iOS 还需要各自平台工具，不能把“有导出选项”写成“已经真机通过”。

## 8. 一个新成员可以领取什么

- 给某个领域脚本补一条测试，并说明规则意义。
- 在 Linux 电脑运行 `test_all.sh`，记录系统版本和结果。
- 检查主场景在不同窗口大小下是否溢出。
- 按工具链文档准备 Android SDK，但不要提交 SDK 文件。

交付时提供命令、结果、环境和 commit。遇到失败先保留完整错误，不要只说“Godot 打不开”。
