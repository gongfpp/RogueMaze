# 第 07 章：先把内容放进表，再让程序读取

这一章适合策划、关卡设计实习生和刚开始写程序的人。目标不是学习 JSON 的全部语法，而是理解为什么“改一张牌”不应该到处修改代码。

## 1. 什么叫数据驱动

写死在流程里的做法是：“如果第二关，放一个尖刺；如果第三关，再放一块 8 秒落下的石头”。数据驱动则把落石写成：

```json
{"type": "FALLING_ROCK", "position": [5, 2], "timer": 8.0}
```

程序只负责“落石怎样工作”，内容文件负责“哪一关、哪个位置、几秒落下”。增加同类落石不用复制程序流程。

数据驱动不是“不写代码”。新机制第一次出现仍然要写程序；同一种机制的关卡参数应该由数据组合。

## 2. 本项目的三层分工

- `game/content/data/gameplay.json`：人编写的内容，像一张严格一些的表格。
- `ContentRegistry`：收货员，检查内容是否完整，把 `LEFT` 等文字翻译成规则值。
- `GameSession` 和规则类：使用已经验收的内容，不负责猜测错别字是什么意思。

如果要把第二关尖刺伤害从 1 改成 2，只改 JSON 的 `damage`。如果要新增“喷火器”，则先由程序成员实现喷火器规则和测试，再由内容成员把它放进节点。

## 3. JSON 最少要知道什么

对象用花括号，数组用方括号，字段名和文字要用双引号：

```json
{
  "id": "spike_gallery",
  "hazards": [
    {"type": "SPIKES", "position": [3, 2], "damage": 1}
  ]
}
```

常见错误是少逗号、多逗号、使用中文引号、ID 拼写不一致。编辑后先跑测试，不要等到手机包里才发现。

## 4. 为什么还要校验

JSON 能成功打开，只说明括号和逗号大致正确，不说明内容能玩。例如两张牌使用相同 ID、方向写成 `RIGTH`、奖励引用不存在的牌、陷阱坐标在棋盘外，或中间节点没有奖励。

`ContentRegistry` 会收集这些问题。一次显示多项错误，比修一个、重启、再看到下一个更适合内容制作。

## 5. 实习生可以领取的安全任务

第一次内容任务可以是“新增一个不引入新机制的节点变体”：

1. 从 `main` 拉取最新代码并新建分支。
2. 只编辑 `gameplay.json`。
3. 不改已有 ID；为新节点使用小写英文和下划线。
4. 运行 `./scripts/test_all.ps1`。
5. 启动游戏，确认文字、牌序和陷阱时机。
6. 提交时写清改了什么、怎样验证、有没有软锁风险。

交付不只是“一份 JSON”。测试通过记录和一段真实试玩结论，也是任务的一部分。

## 6. 怎么判断内容改对了

自动测试回答“格式和已知规则有没有坏”。人要继续回答：给出的牌真的能处理陷阱吗？固定循环手牌会不会连续出现四张无法前进的牌？玩家第一次看到危险时有时间理解吗？这关和上一关的决策有什么不同？

数据文件降低了修改成本，但不会自动产生好关卡。可玩性仍然来自设计、试玩、记录和迭代。
