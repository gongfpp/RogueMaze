# 第 15 章：结算数据不是为了监视玩家

读完这一章，你能解释 RogueMaze 结算页上的数字从哪里来，能用它们辅助一次试玩，也知道什么时候不能只看数字下结论。

## 为什么要做局后复盘

试玩观察员很容易写下“玩家好像经常点错”。这句话无法比较，也很难交给程序员。现在结算页会显示：

- `TIME`：没有暂停时的本局时间。
- `ROADS`：规则接受的铺路次数。
- `MISPLAYS`：玩家真的提交、但被规则拒绝的放置次数。
- `DMG / HEAL`：实际损失和回复的生命，不是陷阱出现次数。
- `DECK + / ARMOR / CUT`：加牌、升级和移除各选了几次。

这些数据让描述更具体。例如“3 位新人分别有 2、9、11 次无效放置，而且后两位都说不理解端口”，比“操作可能有问题”更容易变成待办。

## 先分清生命周期

**生命周期**就是一份数据应该从什么时候存在到什么时候消失。

RogueMaze 中：

- 一格道路只属于当前节点；进入下一节点就清空。
- 生命、远征牌组和复盘统计属于整次远征；换节点时保留。
- 点击 `PLAY AGAIN / TRY AGAIN` 开始新远征；统计归零。

所以统计在 `GameSession.reset()` 清空，不在 `_start_node()` 清空。把代码放错一个函数，结算就可能只显示最后一关的数据。

## “无效放置”怎么数才公平

鼠标经过棋盘时，游戏会不断调用合法性预览。如果每次红色预览都加一，玩家只是移动鼠标就会制造几百次“错误”。

本项目只在 `place_selected()` 收到一次真实点击或拖放结束时计数，而且要求游戏仍在进行、没有暂停。这里的原则是：指标必须对应玩家做出的明确动作。

## 自己验证一次

1. 启动游戏，故意把一张路放到起点格，看到“occupied”。
2. 正常放一张路，然后暂停几秒再恢复。
3. 故意断路失败。
4. 结算页应显示 `MISPLAYS 1`、`ROADS 1`；暂停的几秒不应进入 `TIME`。
5. 点击 `TRY AGAIN` 再失败，新一局数字应从零开始。

自动检查可以运行：

```powershell
.\scripts\test_all.ps1
```

其中规则测试检查单项计数，耐久测试重复 250 次确认跨局不会残留。

## 试玩时怎么记录

每位玩家结束后，抄下结算的两行数据，再问三个短问题：

1. 哪一次失败最意外？
2. 哪张牌最难判断方向？
3. 如果再玩一局，你会改变什么？

数字告诉你“发生了多少次”，回答告诉你“为什么发生”。两者冲突时先回看过程，不要用数字压过玩家的真实理解。

## 隐私与边界

当前统计不保存、不联网，也没有账号。以后若要保存试玩记录，必须先说明用途、保存多久、谁能看，并允许测试者拒绝。姓名、设备标识、精确位置与玩法复盘无关，不应顺手收集。

最后，`MISPLAYS 0` 不能证明界面优秀：玩家也可能因为害怕犯错而完全不尝试。数据是提问的起点，不是替团队替玩家下结论的机器。
