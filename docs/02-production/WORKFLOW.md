# 项目管理与质量流程

## 1. 管理模型

项目使用轻量的迭代方式：路线图控制方向，Backlog 控制顺序，小工作包控制实际开发范围。文档与代码放在同一 Git 仓库。完成一个包并验证后，可以在同一轮继续下一个；阶段门禁只判断质量，不制造等待。

## 2. 单工作包流程

```text
READY → ACTIVE → VERIFY → DONE
           ↓
        BLOCKED
```

### 就绪定义（Definition of Ready）

- 目标能用一句话说明。
- 范围内/范围外明确。
- 依赖已完成或有可执行替代方案。
- 验收标准可观察、可测试。
- 工作量适合一次协作完成；否则先拆分。

### 完成定义（Definition of Done）

- 验收标准全部通过。
- 相关自动测试、手工检查或试玩证据已记录。
- 没有新增的 P0 已知缺陷。
- Backlog、CHANGELOG 和当日开发日志已更新。
- 架构/策划变化已同步，重要取舍形成 ADR。
- 新知识已写入或链接到实战教学。

## 3. 每次开发的标准步骤

1. **领取**：选择一个 READY 工作包，复核验收标准与依赖。
2. **设计**：列出最小方案、测试点与风险；重大方案写 ADR。
3. **实现**：保持小步提交，不加入工作包以外的功能。
4. **验证**：自动测试优先，随后构建、手测或真机试玩。
5. **复盘**：记录做了什么、证据、偏差、技术债与新发现。
6. **版本化**：更新状态和文档，创建 Git commit 并推送远端。
7. **继续或收尾**：资源允许就领取下一个 READY 工作包；遇到自然检查点或真实阻塞再交付总结。

## 4. 版本与分支

- `origin` 固定指向 `https://github.com/gongfpp/RogueMaze.git`。
- `main` 始终保持可运行或仅含通过检查的文档。
- 功能分支建议 `feature/WP-012-auto-runner`。
- 修复分支建议 `fix/WP-xxx-short-name`。
- 提交格式建议：`type(scope): description [WP-xxx]`。
- 类型：`feat`、`fix`、`test`、`docs`、`refactor`、`chore`。
- 里程碑版本：原型 `0.1.x`、垂直切片 `0.2.x`、Alpha `0.3.x`、Beta `0.8.x`、候选 `0.9.x`、首发 `1.0.0`。

每轮开发至少提交一次并推送。提交前检查工作树，只纳入本轮相关文件；不自动执行删除远端分支、覆盖历史、发布商店版本等破坏性或不可逆操作。

## 5. 测试金字塔

- 纯逻辑单元测试：网格、端口、卡牌、随机种子、存档迁移。
- 场景/集成测试：角色通行、陷阱、节点切换、暂停恢复。
- 端到端冒烟：安装、首局、结算、重开、退出恢复。
- 试玩测试：乐趣、认知负担、失败可解释性。
- 平台矩阵：至少一台低/中端 Android、一台受支持 iPhone、Windows 10/11 和一个主流 64 位 Linux 环境；具体版本在 Beta 前锁定。

每个缺陷记录：复现步骤、期望、实际、环境、严重度和证据。

## 6. 质量预算（初始值）

- 桌面原型逻辑测试必须全绿。
- 主流程阻断缺陷为 0 才能过阶段门禁。
- 目标帧率 60 FPS；低端降级 30 FPS，帧率策略需一致。
- 运行时不依赖网络也能完成主线远征。
- 存档写入采用临时文件 + 原子替换，并保留可迁移版本号。

性能的具体毫秒、内存与包体预算在首个真机构建后以测量值校准。

## 7. 文档维护责任

| 发生的变化 | 必须更新 |
| --- | --- |
| 玩法或范围 | GDD、MVP、Backlog、CHANGELOG |
| 架构或依赖 | ARCHITECTURE、ADR、风险登记 |
| 完成工作包 | Backlog、开发日志、CHANGELOG |
| 新测试发现 | 开发日志、GDD 待验证问题或缺陷记录 |
| 可复用知识 | `docs/05-tutorial` 对应章节 |

## 8. 决策与范围控制

- 可逆且局部的实现细节由当前工作包直接决定并记入日志。
- 影响平台、引擎、存档、核心规则或三天以上返工的决定必须写 ADR。
- 新点子默认进入 IDEA，不打断 ACTIVE 工作包。
- 若工作包超过预期两倍，先拆分或降范围，不让一个无法验证的大任务长期占着进行中状态。

## 9. 写给人的文档

- 开头先说读者能完成什么，再讲背景。
- 第一次出现的术语立即用日常语言解释，并给一个本项目里的例子。
- 能用截图、命令或三步操作说明的，不堆抽象原则。
- 记录真实限制和失败方案，不把计划写成已经实现的功能。
- 教程要让零基础实习生能运行、修改、测试并把模块交给下一位成员。
- 文档评审先问“新人照着能不能做成”，再看格式是否漂亮。

## 10. 每轮交付报告格式

- 完成：实际交付和对应工作包。
- 验证：运行的测试/构建/试玩及结果。
- 变更：关键文件与设计取舍。
- 风险：新增或变化的风险。
- 版本：commit 哈希、分支和推送结果。
- 下一步：已经继续的工作包，或唯一推荐的后续工作包。
