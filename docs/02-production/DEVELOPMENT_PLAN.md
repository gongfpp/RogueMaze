# 开发计划与里程碑

本计划以单人/小团队为基线，采用两周迭代。每个工作包仍保持小而可验收，但一次协作在前一个工作包验证通过后可以继续下一个，直到遇到自然检查点或真实阻塞。时间估算用于排序和暴露风险，不视为固定发布日期。

## 1. 总体路线

| 阶段 | 参考周期 | 核心产出 | 退出条件 |
| --- | ---: | --- | --- |
| P0 立项 | 1 周 | GDD、MVP、路线图、风险、流程 | G0 文档基线通过 |
| P1 核心原型 | 2–4 周 | 无引擎模拟、可玩灰盒 | G1 核心循环可玩 |
| P2 垂直切片 | 4–6 周 | 单局完整体验、代表性音画 | G2 陌生玩家可独立完成一局 |
| P3 Alpha | 6–10 周 | MVP 功能与内容齐全 | G3 主流程无阻断 |
| P4 Beta | 4–6 周 | 四平台适配、性能、教学、合规 | G4 平台质量门禁通过 |
| P5 发布准备 | 2–4 周 | 商店、监控、发布候选 | G5 可提交审核 |

总计约 19–31 周，不含商店审核等待、外包资产排期和重大方向返工。

## 2. 工作分解

### P0：立项与预制作

- `WP-000` 策划与管理基线（本轮）
- `WP-001` 无引擎规则原型：模拟卡牌循环、连通性和胜负
- `WP-002` 引擎版本锁定、许可证与移动导出可行性验证

### P1：核心原型

- `WP-010` 创建最小 Godot 工程、测试框架与调试场景
- `WP-011` 网格坐标、路块数据与端口连接单元测试
- `WP-012` 角色沿已连接道路自动移动
- `WP-013` 触屏拖拽、旋转、幽灵预览和合法性反馈
- `WP-014` 手牌、牌库、弃牌与循环
- `WP-015` 胜利、失败、暂停与重开闭环
- `WP-016` 第一轮可玩性测试与规则调整

### P2：垂直切片

- `WP-020` 关卡骨架和镜头滚动
- `WP-021` 尖刺与落石
- `WP-022` 三节点流程和奖励选择
- `WP-023` HUD、教学和失败复盘
- `WP-024` 纸片/机械剧场视觉试制与基础音效
- `WP-025` Windows/Linux 桌面包与 Android/iOS 真机包
- `WP-026` 5 人可用性测试和 G2 评审

### P3：Alpha

- 数据驱动卡牌、陷阱、角色与关卡配置
- 工程师基础牌组及升级/移除逻辑
- 三类关卡骨架、事件节点、终点挑战
- 设置、存档、解锁与版本迁移
- 调试菜单、日志和自动化测试扩展

### P4：Beta

- 设备分级、性能预算、对象复用与加载优化
- 分辨率、安全区、刘海/圆角、窗口缩放、鼠标键盘、触觉与音频中断适配
- 新手引导、可访问性、文本本地化准备
- 崩溃恢复、存档破坏测试、长时间稳定性测试
- 隐私、第三方 SDK、商店政策与年龄分级评审

### P5：发布准备

- 商店文案、截图、图标、预览素材与测试轨道
- 发布候选构建、回归测试、已知问题和回滚方案
- 支持渠道、隐私政策、版本节奏与首周监控计划

## 3. 迭代节奏

- 周期开始：从 Backlog 领取已满足就绪定义的工作包。
- 开发中：先写验收/测试，再实现最小闭环；范围变化写入决策记录。
- 周期结束：构建、测试、试玩、更新日志和教程，评审是否过门禁。
- 每轮至少一次 Git commit 并推送远端；长迭代在可运行的检查点增加提交。
- 已完成且验证通过时可以继续领取下一个 READY 工作包，无需等待形式化批准。
- 每 2–3 个迭代进行一次范围与风险复盘。

## 4. 首三个迭代

### Iteration 0：策划基线

目标：建立统一产品方向和工作方式。交付 `WP-000`，不产出游戏代码。

### Iteration 1：规则证明

目标：用纯逻辑模拟验证路块连接、固定循环/洗牌循环和基本可解性。交付 `WP-001`，输出可重复测试和一份试玩结论。

### Iteration 2：引擎骨架

目标：在锁定工具链后创建最小工程，在桌面运行网格与单元测试。交付 `WP-002`、`WP-010`、`WP-011`，移动导出只做冒烟，不扩展玩法。

## 5. 资源与依赖检查点

- P1 前：确认目标 Godot 稳定版本、Git LFS 是否需要、Windows/Linux 导出以及 Android SDK/JDK 环境。
- P2 平台构建前：确认 Apple 开发环境、签名账号、Android/iOS 真机和 Linux 测试环境。
- P3 美术量产前：锁定分辨率、像素密度、动画方式与资产命名规范。
- P4 SDK 接入前：完成隐私影响和包体/性能评审。
