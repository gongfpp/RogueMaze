# 5 人 Alpha 试玩执行指南

## 这次要回答的问题

1. 陌生玩家能否在 2 分钟内理解“选路牌、旋转、放到绿色格”？
2. 玩家第一次失败后，能否说出失败原因并尝试不同方案？
3. ADD、ARMOR、REMOVE 是否能让玩家做出有理由的选择？
4. 尖刺、蒸汽、落石和维修支路是否能在造成后果前被看懂？
5. 一次五节点远征的节奏是否愿意让玩家再来一局？

## 招募与隐私

- 至少 5 人；不能是参与代码、策划或美术的人。
- 用 `P01`、`P02` 编号，不记录姓名、账号、联系方式或设备唯一标识。
- 开始前说明这是未完成版本，会记录操作观察和玩家主动说出的意见；不录音录像，除非另行取得明确同意。
- 玩家可以随时停止，不需要解释原因。

## 每场流程（约 20 分钟）

1. **2 分钟准备**：确认 ZIP、SHA-256、Build 身份、窗口和声音正常。
2. **8 分钟首局**：只读 `PLAYER_README.txt`，主持人不解释端口、路线和奖励答案。请玩家边玩边说看到什么、想做什么。
3. **3 分钟追问**：问观察表中的三个固定问题，不把自己的结论塞进问题。
4. **5 分钟第二次尝试**：只允许解释玩家明确问到的操作，不代替做构筑决策。
5. **2 分钟收尾**：抄录结算数据和 Build，确认玩家是否愿意再次参加；不要索取与本次测试无关的信息。

## 主持人可以说什么

- 可以：“请先按你自己的理解试一次。”“你刚才期待发生什么？”
- 不可以：“这里应该上坡。”“ARMOR 肯定更好。”“其他人都看懂了。”

帮助过早会把教学问题伪装成成功；沉默也不是冷漠，主持人仍要在程序卡死或玩家不舒服时立即中止。

## 完成门禁

- 5 份完整观察表，均有 Build、环境、首局结果和固定问题回答。
- 将问题按 P0 阻断、P1 明显摩擦、P2 建议归类；相同根因合并，不按玩家人数机械创建重复缺陷。
- P0 必须修复并用同一复现步骤回归；P1 明确处理或记录不改理由。
- 更新 GDD 的待验证问题、Backlog、开发日志和试玩结论。没有这些证据，不把 WP-016/WP-026 改成 DONE。
