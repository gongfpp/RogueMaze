# 技术架构草案

状态：工程基线 v0.1。目录和核心规则已经落地，后续按真实实现继续修订。

## 1. 技术方向

- 引擎：Godot 4.7 stable 标准版，已锁定并完成桌面导出。
- 语言：GDScript，用于快速迭代；性能热点测量后再考虑原生扩展。
- 目标：Android、iOS、Windows、Linux 共用规则和内容层。
- 画面：2D，逻辑网格与视觉表现解耦；移动与桌面使用可替换的输入映射和响应式界面布局。
- 配置：资源文件或可校验的数据对象；卡牌、陷阱、关卡不把数值散落在场景脚本中。
- 随机：所有影响玩法的随机数从单局种子派生，支持复现失败和测试。

选择依据与替代条件见 [ADR-0001](decisions/ADR-0001-engine-and-stack.md)。

## 2. 分层

```text
UI / Input
    ↓ command
Game Session Orchestrator
    ↓
Pure Domain Rules ─── Content Definitions
    ↓ events
Presentation / Audio / Haptics
    ↓
Platform Services (save, lifecycle, analytics opt-in)
```

- **Domain**：网格、端口、路块、牌组、角色状态、胜负与种子随机。尽量不依赖 Godot 节点，便于测试。
- **Application**：管理一局的阶段、命令与事件，协调 Domain。
- **Presentation**：场景节点、动画、相机、音效和反馈，不决定规则。
- **Infrastructure**：存档、移动端生命周期、桌面窗口、文件路径、构建和可选服务。

## 4. 核心数据模型草案

- `GridPosition(x, y)`：不可变网格坐标。
- `Direction`：左/右/上/下，提供反方向映射。
- `RoadDefinition`：卡牌静态定义、端口和关键词。
- `PlacedRoad`：定义引用、坐标、旋转、耐久和运行时状态。
- `BoardState`：边界、占用与放置/连接查询。
- `DeckState`：抽牌堆、手牌、弃牌堆、循环策略。
- `RunnerState`：所在路段、进度、生命和状态效果。
- `RunState`：种子、节点、资源、胜负和版本。

UI 只发送 `PlaceRoadCommand`、`RotatePreviewCommand` 等意图；规则层返回成功事件或带原因的拒绝结果。

## 5. 关键技术原则

- 确定性：相同版本、配置、输入和种子应得到相同逻辑结果。
- 数据驱动：新增普通卡牌尽量不改流程代码。
- 可观察：开发构建可查看牌堆、种子、网格端口和状态事件。
- 多平台输入：触摸、鼠标和键盘都转换为相同的游戏命令，不复制规则代码。
- 平台差异隔离：后台/恢复、安全区、窗口缩放和文件路径留在平台层。
- 无隐式全局状态：全局服务保持最少，单局状态可重建和测试。

## 6. 首批测试边界

- 旋转后端口变换。
- 相邻端口双向匹配。
- 越界、重叠和孤立路块拒绝原因。
- 固定顺序与随机洗回在种子下可复现。
- 角色到达端口时选择下一路段或明确失败。

## 7. WP-001 已验证规则

- 坐标从 `(0, 0)` 开始，`x` 向右、`y` 向下。
- 道路接口使用左、上、右、下四个离散方向，旋转以顺时针 90 度为一步。
- 新道路必须至少连接一块已有道路；相邻道路若只有一侧开口，则明确返回 `PORT_MISMATCH`。
- 越界、重叠、接口不匹配和孤立放置分别返回不同失败原因，方便 UI 给玩家准确反馈。
- 路径连通使用图搜索判断，不让画面节点决定胜负。
- 固定循环与弃牌洗回共用一个牌组接口；后者从单局种子派生随机顺序。

当前实现位于 `prototypes/rules`，是 Node.js 可执行规格。`WP-011` 会把同样的规则和测试样例移植到 Godot/GDScript；正式游戏不依赖 Node.js。

`WP-011` 已完成移植。Node 规格保留为独立参考，GDScript 是正式运行时实现；两边目前分别通过 15 个测试和 30 项断言。

## 8. 首个可玩闭环

- `RunnerState` 只读取 `BoardState`：选择已双向连接且不是来路的下一格，优先向右，再考虑上下和回退。
- 断路时角色在格子中心等待 1.75 秒；玩家补路后继续，否则进入 `FAILED`。
- `GameSession` 管理棋盘、角色、四张手牌、倒计时、暂停和胜负，不负责画图。
- `main.gd` 把触摸、鼠标和键盘转换成选牌、旋转、放置、暂停命令，并用几何图形画出灰盒。
- 点击与拖拽共用 `place_selected`，不会出现两套放置规则。
- 程序化完整单局会铺满 7 格平路并让角色抵达终点，防止系统各自正确却串不起来。

## 9. 垂直切片远征

- 一次远征由 `GameSession.node_index` 驱动三个短节点，生命和 `run_deck` 跨节点保留，棋盘/角色/手牌每节点重建。
- 节点 1 教学基础拼路；节点 2 加尖刺；节点 3 同时保留尖刺并加入定时落石。
- 尖刺在角色第一次进入时扣 1 点生命；落石计时结束后从 `BoardState` 移除道路。
- 每个节点胜利后进入 `REWARD`，选择的道路 ID 加入本次远征牌组，再进入下一节点。
- 失败原因由应用层记录为断路、落石或生命耗尽，表现层只负责翻译成玩家反馈。
- 测试完整跑过三节点，并单独验证落石能摧毁未来道路。

## 10. 表现事件与程序音频

- `GameSession` 将节点完成、总胜利、失败、尖刺和落石写入短生命周期事件队列。
- `main.gd` 每帧消费事件并播放表现，不让规则层引用 AudioStream 或动画节点。
- `AudioCues` 运行时生成 22.05 kHz、16-bit、单声道短音，不需要外部音频文件；四个播放器允许提示音短暂重叠。
- 纸片机械背景按目标宽高比等比中心裁切，再叠加深色蒙层。道路和危险仍由程序几何图形绘制。
- 原始背景约 2.7 MB；Godot 导入压缩后，Windows 与 Linux 发布包均增加 1,982,432 字节。后续背景版本以这一增量作为预算基线。

## 11. 本地持久化

- `PersistenceService` 位于平台层，领域规则不知道文件路径。
- `settings.json` v1 保存 SFX 和减少动态效果；`progress.json` v1 保存胜场、最佳节点和已解锁道路。
- 加载数据先经过类型、范围、重复项和已知道路 ID 校验；版本缺失按旧版迁移，无法解析则回到安全默认值。
- 保存先写 `.tmp` 并 flush；旧文件改名为 `.bak`；临时文件改名为正式文件；成功后删除备份。替换失败时尝试恢复备份。
- 测试使用独立 `user://tests` 文件，不触碰玩家正式数据，并在测试结束清理临时/备份文件。
- UI 在节点完成和总胜利事件发生时更新进度；设置按钮修改后立即保存。

## 12. 内容注册表

- `game/content/data/gameplay.json` 是道路静态定义、初始牌组和节点编排的唯一作者来源。
- `ContentRegistry` 在内容进入玩法前完成结构与引用校验；规则层只接收已经转换的 `RoadDefinition`、节点和陷阱数据。
- 内容文件带 `schema_version`。字段结构发生不兼容改变时提升版本并给出明确错误，不能悄悄猜测旧字段。
- 道路 ID 是存档和牌组引用使用的稳定键；显示名称可以修改，已经发布的 ID 不改名、不复用。
- 新增已有机制的节点只编辑内容；新增陷阱行为仍需先实现规则、事件和测试，再让内容引用。

## 13. 卡牌实例与构筑动作

- `RoadDefinition` 是不可变的道路种类；`CardState` 是本次远征拥有的具体牌，包含稳定实例 ID、道路 ID 和等级。
- `DeckState` 只负责循环/洗回任意卡牌值，不理解强化效果；`GameSession` 负责创建实例和应用奖励。
- 奖励项保存目标实例 ID，不保存易受删牌和排序影响的数组下标。
- 当前强化上限为 2。2 级道路吸收一次落石后回到 1 级；陷阱写入 `neutralized`，规则和表现读取同一结果。
- UI 用 `+`、`ARMOR` 和棋盘外圈重复编码等级，不能只依赖颜色。

## 14. 五节点与周期陷阱

- 远征节点仍由内容数组顺序驱动；应用层只维护当前下标，五节点不是五套场景脚本。
- 蒸汽喷口用 `elapsed + phase` 对 `cycle` 取余，和 `active_duration` 比较得到开关状态。
- 规则扣血、UI 颜色/形状和切换倒计时调用同一个开关判定，避免画面与碰撞窗口不同步。
- 最终压力核心只组合已教学的尖刺、蒸汽和落石，不在终点临时引入第四种未知规则。

## 15. 平台边界与安全区

- 四个平台共用同一主场景和规则；差异放在导出预设、输入映射、文件服务和构建脚本。
- Android/iOS 布局把系统安全矩形换算到逻辑视口，再在矩形内计算棋盘、手牌和按钮；桌面使用完整窗口。
- 导出预设不保存 keystore、密码、证书、描述文件或 Team ID。发布脚本从受控环境临时注入，并在退出时恢复仓库占位。
- 当前游戏不联网、不振动、不读相册/联系人/位置，因此 Android 预设不申请这些权限，iOS 不添加对应隐私声明。

## 16. 生命周期与干净环境验证

- 应用暂停、窗口失焦和 Android 返回请求调用幂等 `set_paused(true)`；不能用连续通知会反转状态的 toggle。
- 离开应用和场景退出都会保存设置与进度；两份写入分别执行后再合并成功结果。
- GitHub Actions 的规则测试不依赖导出模板；Linux artifact 任务单独缓存锁定模板，避免每个 PR 下载 1 GB 以上文件。
- CI、本地构建都同时检查退出码、失败日志词和最终文件存在/非空，抵御 Godot 导出失败返回 0 的情况。

## 17. 棋盘收益要素

- `REPAIR_PAD` 与陷阱共用数据化坐标和进入格触发管线，但产生正向 `REPAIRED` 事件。
- 维修只在生命未满时消耗，回复结果夹在最大生命内；成功后 `spent` 阻止重复利用。
- 表现层使用颜色、圆环、加号和节点短提示表达，不通过血量差分猜测治疗来源。

## 18. 发行署名与许可证

- `assets/legal` 保存 Godot MIT 正文、锁定 `4.7-stable` 的完整 `COPYRIGHT.txt` 和项目 Credits。
- 四平台导出预设用 include filter 强制携带 `.txt` 声明，不能依赖 Godot 是否把未引用文本自动识别成资源。
- 暂停界面提供引擎与许可证摘要；完整清单随包分发。升级引擎时版权文件和哈希必须一起更新。
- 项目自身代码采用何种对外许可证属于所有者决策；在负责人明确前不自动添加开源 LICENSE。
